## 전일 예측 검증

2026-06-12 기준 예측은 2026-06-15 시장을 대상으로 했다. 당시 모델은 반도체/전자를 핵심 관찰로 두고, 자동차, 금융, 조선/방산, 2차전지를 보조 관찰로 보았다.

| 섹터 | 전일 행동 라벨 | 실제 수익률 | 검증 |
| --- | --- | ---: | --- |
| 반도체/전자 | 핵심 관찰 | -0.17% | 실패 |
| 자동차 | 보조 관찰 | +5.52% | 성공 |
| 금융 | 보조 관찰 | +3.55% | 성공 |
| 조선/방산 | 보조 관찰 | +6.63% | 성공 |
| 2차전지 | 보조 관찰 | +5.11% | 성공 |

핵심 관찰 1순위였던 반도체/전자는 빗나갔다. 하지만 보조 관찰 4개 섹터는 모두 양수 수익률을 기록했고, 실제 상위권에 들어왔다. 오늘의 핵심은 모델이 시장의 반등 후보군은 비교적 잘 잡았지만, 가장 강한 섹터를 반도체/전자로 본 점은 틀렸다는 것이다.

## 모델이 학습한 점

- 급반등장에서는 핵심 관찰 1개보다 보조 관찰 후보군 전체의 품질이 더 중요할 수 있다.
- 반도체/전자는 직전 강한 랭킹 신호가 있었지만, 실제 당일에는 순환매의 중심이 조선/방산, 자동차, 2차전지로 이동했다.
- 장중 반등 모델은 2차전지 강세를 비교적 빨리 포착했고, 일봉 예측의 보조 관찰 신호와도 방향이 맞았다.
- 보정 확률은 여전히 낮은 신뢰도로 나오고 있어, 강한 상승 후보라도 최종 판단은 공격보다 관찰 중심으로 유지해야 한다.
- 포트폴리오 관점에서는 예측 성공만 보여주는 것보다, 핵심 관찰 실패와 보조 관찰 성공을 함께 기록하는 편이 더 설득력 있다.

## GitHub 포트폴리오 튜닝

오늘은 GitHub 포트폴리오 자료와 음성 자료를 기준으로 README 구조를 다시 정리했다. 자료에서 반복적으로 강조된 핵심은 다음과 같다.

- 첫 화면에서 프로젝트 목적, 본인 역할, 결과, 증거 링크가 보여야 한다.
- 단순히 기능을 나열하지 말고 어떤 문제를 해결했는지 보여줘야 한다.
- 문제를 발견한 과정, 선택한 해결책, 포기한 대안, 검증 결과가 기록되어야 한다.
- Markdown 표, 링크, 코드 블록, Mermaid 흐름도를 사용해 읽는 사람이 빠르게 판단할 수 있게 해야 한다.
- 일일 일기는 결과만 쓰지 말고 실패와 수정 과정을 같이 남겨야 한다.

이에 맞춰 README를 `5초 요약`, `전체 파이프라인`, `모델 구조`, `최신 모델 성능`, `일일 예측 일기`, `문제 해결 기록` 중심으로 재구성했다. 또한 오류와 개선 과정을 `docs/problem-solving-log.md`에 따로 누적하도록 만들었다.
